Обучение Git

  • Git – популярная система для хранения программного кода, (система контроля версий)

    Запомнить! Система контроля версий (CVS - Control Version System) - позволняет вернуться к определённой версии файла или определить, кем и когда были сделаны изменения в файле.

    Запомнить! Репозиторий Git – хранилище для истории разработки проекта, которое содержит все версии используемых файлов.

    Запомнить! Коммит (commit) - фиксация состояния проекта в репозитории Git

    Запомнить! Индекс – список файлов, изменения в которых система Git отслеживает. В индексе также предварительно сохраняются все изменённые файлы, которые будут зафиксированы в репозитории при следующем коммите

    Запомнить! Ветка – изолированный независимый поток разработки, в котором можно делать коммиты, недоступные для других веток

    Основные возможности git:

    1. Добавлять новый функционал.
    2. Исправлять ошибки.
    3. Работать в команде.

  • Каждый Git-проект состоит из трёх областей: рабочий каталог, индекс и репозиторий.

    1. Рабочий каталог - текущие файлы над которыми мы работаем. Когда мы создаем новые файлы или изменяем уже существующие, то эти файлы сохраняются в рабочем каталоге. Рабочий каталог не имеет отношения к папке .git.
    2. Индекс находится в папке .git и содержит информацию о том, какие именно файлы и в каком состоянии будут зафиксированы при следующем коммите.

      Изменим файл style.css и пропишем git add .

      Тем самым добавим файл в группу индекс (индекс содержит файлы готовые для коммита).

    3. HEAD - текущий слепок на момент последнего коммита (это слепок последних закомиченных изменений)

    Обработка файлов в Git происходит следующим образом:

    1. Мы редактируем файл в рабочем каталоге. У такого файла будет состояние "Изменен" (modified).
    2. Индексируем изменённый файл, добавляя его в индекс командой git add. Статус такого файла – "Индексирован" (staged)
    3. Сохраняем изменения в репозитории, выполняя коммит с помощью git commit. Статус такого файла – "Зафиксирован" (fixed).

    Посмотреть содержимое файла в консоле можно с помощью команды cat ...

    Отменение изменений

    Что-бы откатиться к нужному коммиту существует команда git reset с параметрами --soft и --hard

    git reset --soft 'id' переместит указатель HEAD на нужный коммит, при этом рабочий каталог и индекс остаются прежними (соответствующими коммиту, с которого мы ушли). (Изменения которые произошли после коммита к которому хотим перейти останутся, можно будет открыть нужный файл, произвести в нём изменения, и затем закоммитить, и будет создан следующий коммит).

    git reset --hard 'id' позволяет полностью отменить все изменения, которые были сделаны после определённого коммита. (удаляет файлы и полностью возращает состояние проекта к текущему коммиту к которому мы хотим откатится).

    Запомнить! Не обязательно копировать полностью идентификатор (будет достаточно 7 символов или больше)

  • git branch - Проверить какие ветки уже есть в нашем проекте.

    Как видим есть одна основная ветка с именем master.
    Ветки при разработке можно называть как угодно, но master – это общепринятое имя для основной ветки проекта, в которую добавляется функционал, разработанный в других ветках

    Новый функционал разрабатывают в новых ветках (они называются тематическими), изолированно от основной ветки проекта. Когда все нужные изменения сделаны, функционал полностью готов и протестирован, тематическая ветка объединяется (сливается) с основной и новая версия (релиз) проекта выкладывается на удалённый сервер.

    Создание и переключение веток

    git checkout -b имя_ветки - создаёт новую ветку и сразу переходит в неё.

    Имя для ветки нужно выбирать осмысленно, чтобы было понятно ее назначение (реализация нового функционала или исправление выявленных ошибок)

    Например нужно сделать корзину для покупок, поэтому новую ветку можно назвать feature-cart

    git checkout имя_ветки - переключение между ветками

    Все ветки существуют параллельно друг другу, разработка в них ведётся независимо

    git log --oneline - git log в компактном варианте.

    Слияние веток

    Предположим, что вся работа в ветке feature-cart выполнена, нужный функционал разработан и протестирован. Теперь нам нужно слить ветку feature-cart с основной веткой master. Для этого переключимся в ветку, в которую будем добавлять изменения (то есть в ветку master):

    git merge имя_ветки - Слияние веток.

    Удаление веток

    После того, как произошло слияние двух веток, ветка feature-cart больше не нужна:

    git branch -d имя_ветки - Удаление ветки (d - delete)

  • Если изменения в ветках затрагивали разные строки одного файла, то скорее всего Git сможет слить эти файлы без ошибок.

  • Иногда возникает необходимость оперативно переключиться в другую ветку, но в текущей рабочей ветке файлы еще не готовы для коммита. В этом случае Git позволяет сохранить сделанные изменения, не создавая при этом коммит.

    git stash - собирает незакоммиченные изменения, удаляет их из файлов и в специальном виде архивирует их в Git (помещает в стек незавершенных изменений).

    git stash list - посмотреть что находится в stash

    git stash apply - добавит последний добавленный stash

    git stash apply stash@{0} - добавит конкретный stash

    git stash pop - добавит последний stash и сразу удалит из списка stash

    git stash drop stash@{0} - удалит запись из stash

    Например, в ветке bug-cart что-то разрабатывается, и срочно нужно что-то поменять в другой ветке:

    1. Выполняем команду git stash
    2. Переключаемся к нужной ветке, делам работу которую требуют. (добавляем в индекс, в коммит...)
    3. Возращаемся назад к ветке где архивировали файлы (git stash)
    4. Добавляем stash
    5. Удаляем этот stash из списка

    1. Создание нового репозитория на GitHub
    2. Загрузка локального репозитория на GitHub

      Чтобы загрузить локальный репозиторий на GitHub, нужно сообщить Git'у адрес нужного нам удалённого репозитория

      git remote add origin ...{link}

      origin - имя (может быть любым) У одного проекта может быть несколько удалённых репозиториев под разными именами (разные стадии проекта могут заливаться на разные ресурсы).

    3. git push

      git push удалённый_репозиторий ветка - для выгрузки содержимого локального репозитория в удаленный репозиторий. Она позволяет передать коммиты из локального репозитория в удаленный.

      Ключ -u (upstream) здесь указывает на то, что мы заливаем на сервер ту ветку (master), которой там ещё нет. Во время выполнения команды GitHub запрашивал учетные данные (имя и пароль) для доступа к репозиторию.

      git push -u origin master

    Обязательные файлы на GitHub

    1. .gitignore содержит список файлов и каталогов, за которыми Git не должен следить и которые не будут сохраняться в репозитории.
    2. readme.md - описанием проекта.
    Запомнить! Пустые папки Git игнорирует.
